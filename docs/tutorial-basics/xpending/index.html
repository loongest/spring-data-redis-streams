XPENDING

The XPENDING command shows information about pending entries in a stream for a given consumer group.
A pending entry is a message that was delivered to a consumer but has not yet been acknowledged with XACK.

Syntax​

XPENDING key group [[IDLE min-idle-time] start end count [consumer]]
  • key: The name of the stream.
  • group: The name of the consumer group.
  • IDLE min-idle-time: (Optional) Filters messages that have been idle for at least min-idle-time milliseconds.
  • start: The starting stream ID for the range. - is a special value representing the smallest possible ID.
  • end: The ending stream ID for the range. + is a special value representing the largest possible ID.
  • count: The maximum number of messages to return.
  • consumer: (Optional) Returns only the pending messages for a specific consumer.

Basic usage – check if there are pending messages​

In the simplest form, you can use XPENDING to check whether there are any unacknowledged messages in the group:

XPENDING race:france race-group

Example output:

1) (integer) 20                       ← Total pending (unacknowledged) messages
2) "1754477775952-0"                  ← Oldest pending message ID
3) "1754477775952-19"                 ← Newest pending message ID
4) 1) 1) "consumer-B"                 ← Consumer name
      2) "20"                         ← Number of pending messages for this consumer

This tells us that consumer-B has 20 messages that were delivered but not yet acknowledged.

Extended usage – see idle time and delivery count​

If you want to know how long each message has been pending and how many times it has been delivered, use the extended form:

XPENDING race:france race-group - + 10
1) 1) "1754477775952-0"        ← Message ID
   2) "consumer-B"             ← Consumer name
   3) (integer) 103170         ← Idle time in ms (~103 seconds since last delivery)
   4) (integer) 1              ← Delivery count

Here:

  • Idle time = how long the message has been pending without being acknowledged
  • Delivery count = how many times Redis has delivered this message to consumers

Example: Finding long-idle messages for a specific consumer​

You can combine the filters to find messages that have been idle for a long time for a specific consumer. This is useful for targeting potentially "stuck" jobs.

# Find up to 10 messages for 'consumer-B' that have been idle for at least 60 seconds (60000 ms)
XPENDING race:france race-group IDLE 60000 - + 10 consumer-B

When to use XPENDING​

From experience, there are two major reasons to use XPENDING regularly:

  1. Recovery of unprocessed messages Use XPENDING to check for stuck or abandoned messages. If found, you can recover them with XAUTOCLAIM or XCLAIM so another consumer can process them.

  2. Safe housekeeping before trimming Redis Streams are append-only logs. If your application relies heavily on them, streams can grow very large over time. You can use XTRIM to remove old entries and save memory, but never trim blindly — always check XPENDING first to ensure you’re not deleting unprocessed messages.

In the next topic, we’ll cover how to combine XPENDING with XTRIM to safely manage stream size without losing unprocessed data.
