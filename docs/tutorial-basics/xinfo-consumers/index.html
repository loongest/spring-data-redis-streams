XINFO CONSUMERS

The XINFO CONSUMERS command returns the list of consumers that belong to a specific <groupname> consumer group of the stream stored at <key>.

XINFO CONSUMERS <key> <group>

Sample output:

127.0.0.1:6379> XINFO CONSUMERS race:france race-group
1) 1) "name"
   2) "Chee-MacBook-Pro.local"

Why It Matters​

In real-world applications — especially in microservices — consumer names are often dynamically assigned, not hardcoded. For example, they may be based on the hostname or injected as a random string at deployment time (e.g., in Kubernetes or Docker).

Example: Dynamic Consumer Name via Hostname​

@Configuration
public class BeanConfig
{
    @Bean
    public String hostname() throws UnknownHostException {
        return InetAddress.getLocalHost().getHostName();
    }
}

Consumer Class with Injected Hostname​

We modify our consumer to use the hostname as the consumer name:

@Slf4j
@Component
public class RaceStreamConsumer {

    private final StringRedisTemplate redisTemplate;
    private final RaceStreamRepository repository;
    private final ExecutorService executor = Executors.newSingleThreadExecutor();
    private volatile boolean running = true;

    private static final String STREAM_KEY = "race:france";
    private static final String GROUP_NAME = "race-group";
    // constructor injection
    public RaceStreamConsumer(
            StringRedisTemplate redisTemplate,
            RaceStreamRepository repository,
            @Qualifier("hostname") String consumerName
    ) {
        this.repository = repository;
        this.redisTemplate = redisTemplate;
        this.CONSUMER_NAME = consumerName;
    }

    ...
}

Verifying Consumer Status​

Once your application starts, and your consumer thread begins reading from the stream, you can check the stream contents with:

XRANGE race:france - +

However, if your logs show no message processing activity, how can you be sure your consumer is properly registered? Use XINFO This lets you confirm that the consumer was registered within the group.

XINFO CONSUMERS race:france race-group

Why Messages May Not Be Consumed​

Even if the consumer is registered, it might not consume any messages. One common reason is how the stream is being read:

List<MapRecord<String, Object, Object>> records = redisTemplate.opsForStream().read(
    Consumer.from(GROUP_NAME, CONSUMER_NAME),
    StreamReadOptions.empty().block(Duration.ofMinutes(1)).count(10),
    StreamOffset.create(STREAM_KEY, ReadOffset.lastConsumed())
);

This is equivalent to :

XGROUP CREATE race:france race-group 0 MKSTREAM
XREADGROUP GROUP race-group consumer-B COUNT 10 BLOCK 5000 STREAMS race:france >

Key Behavior to Understand​

  • Messages in a Redis Stream are only delivered once per consumer
  • Messages that are delivered but not acknowledged will remain in a pending state.
  • ReadOffset.lastConsumed() tells Redis to only deliver new messages, not the ones already pending or acknowledged.

How to Handle Unacknowledged Messages​

If you want to temporarily reprocess unacknowledged messages:

StreamOffset.create(STREAM_KEY, ReadOffset.from("0"))

⚠️ But this approach is risky — multiple consumers doing this may result in duplicate processing.

Recommended: Use XCLAIM (or Spring’s .claim())​

Instead of rewinding to 0, the preferred production-safe way is to recover unacknowledged messages via XCLAIM.

We’ve shown how to implement this previously using Spring’s claim() method inside a scheduled job.
